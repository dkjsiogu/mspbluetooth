# 多点温度/环境监测系统报告提纲

## 1. 项目需求分析

- DHT11 采集温湿度。
- MQ-2 采集可燃气体模拟量并分级。
- HC-SR04 采集距离。
- OLED 显示温湿度、气体、距离、阈值和报警状态。
- HC-05 蓝牙发送实时数据到手机，并接收阈值和历史查询命令。
- 内部 Flash 保存历史数据，并支持独立读取。
- 温度和气体超限时蜂鸣器分级报警。
- EC11 或蓝牙命令调整温度报警阈值。

## 2. 总体方案设计

- 主控：MSP430F5529。
- 采集层：DHT11、MQ-2、HC-SR04。
- 交互层：OLED、HC-05、EC11、蜂鸣器。
- 存储层：MSP430 Information Flash。
- 软件层次：应用调度、传感器驱动、显示驱动、蓝牙协议、Flash 日志、报警控制。

## 3. 硬件设计流程

1. 分配引脚并检查冲突。
2. 先连接 3.3 V 数字器件：DHT11、OLED、HC-05、EC11。
3. 对 5 V 或模拟输出器件做电平安全检查：MQ-2 AO、HC-SR04 Echo。
4. 按模块逐个通电测试，最后进行整机联调。

核心引脚：

| 模块 | 引脚 |
| --- | --- |
| DHT11 | DATA=P1.0 |
| MQ-2 | AO=P6.0/ADC12_A0 |
| HC-SR04 | Trig=P1.2，Echo=P1.3 |
| OLED | SDA=P3.0，SCL=P3.1 |
| HC-05 | TX/RX=P4.5/P4.4 |
| 蜂鸣器 | P2.0 |
| EC11 | A=P2.1，B=P2.2，SW=P2.3 |

## 4. 软件设计流程

1. 初始化时钟、GPIO、系统 tick、UART、EC11、传感器、OLED、蜂鸣器和 Flash。
2. 周期采集 DHT11、MQ-2、HC-SR04。
3. 计算温度超限等级和气体等级，取最大值作为综合报警等级。
4. 刷新 OLED 四行显示。
5. 定时通过蓝牙输出 `DATA` 实时数据。
6. 定时把采样快照写入内部 Flash。
7. 解析蓝牙命令，实现实时刷新、阈值调整、历史读取、清空和接线诊断。
8. 处理 EC11 旋转和按键事件，实现本地阈值调整和单次刷新。

## 5. 蓝牙协议设计

| 命令 | 作用 |
| --- | --- |
| `?` | 上传实时数据 |
| `i` | 查询固件信息 |
| `w` | 查询接线 |
| `h` | 查询帮助 |
| `x` | 查询操作轨迹 |
| `T+` / `T-` | 阈值增减 |
| `SETT=32.0` | 设置温度阈值 |
| `HIST?` | 历史数量 |
| `HIST 1` | 读取指定历史 |
| `DUMP` | 输出全部历史 |
| `CLRLOG` | 清空历史 |

## 6. 存储设计

- 使用 Information Flash B/C/D，避开通常保存校准数据的信息段 A。
- 每条记录保存序号、运行时间、温湿度、气体、距离、阈值、报警等级和校验。
- 写满后整体擦除并从新序号继续记录。
- 蓝牙 `HIST n` 和 `DUMP` 可独立读出历史记录。

## 7. 验证流程

- 软件验证：`tools\run_verification.ps1`。
- APK 验证：`tools\verify_android_apk.ps1`。
- 交付打包：`tools\package_release.ps1`。
- 硬件验证：按 `docs\hardware_verification.md` 逐项记录。

## 8. 风险分析

- HC-SR04 Echo 和 MQ-2 AO 可能超过 MSP430 3.3 V 输入范围，必须先分压或限幅。
- DHT11 单总线时序对延时精度敏感，读数异常时先查上拉和中断影响。
- MQ-2 需要预热，预热前不应作为报警依据。
- 内部 Flash 擦写次数有限，日志周期不宜过短。
- 蜂鸣器若为无源器件，需要改用 Timer PWM 驱动。
