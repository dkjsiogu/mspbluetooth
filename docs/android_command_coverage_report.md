## Structured Response Parsers

| Response | APK handler evidence |
| --- | --- |
| `status=...` | `updateDashboard` extracts mode, track, volume, order, progress |
| Dashboard visual bars | `volumeBar` and `progressBar` render bounded status values as 0-32 volume and 0-100 progress bars |
| `sd/info/selftest/tone/open/error` | `updateHealthPanel` renders storage, firmware, self-test, audio-test, file-open, and error evidence |
| `display 1/2/3:...` | `updateDisplayFrame` renders the three-line display model |
| `tracks ...` | `updateTrackList` renders TRACK01..TRACK09 availability |
| `link ...` | `updateLinkPanel` renders RX/status/display/error counters for Bluetooth round-trip evidence |
| `input ...` | `updateInputPanel` renders EC11 and S1/S2/S4 short/long event counters |
| `pin ...` | `updateWiringPanel` renders TF/I2S/EC11/local/BT/e-paper wiring diagnostics |
| Connect bootstrap | `syncInitialPanels` sends `?`, `l`, and `d` after RFCOMM connect |
| Acceptance script | `Run Acceptance` sends diagnostic, display, tone, and control commands with `TX>` log markers |
| Acceptance summary | `acceptanceView` shows SD, info, selftest, tracks, wiring, display, status, tone, and file-open evidence as `Acceptance X/9` |
| Offline phone demo | `Demo RX` injects representative firmware responses into the same parser so Android panels can be checked without HC-05 hardware |
| Acceptance log export | `Share Log` uses Android `ACTION_SEND` with `EXTRA_TEXT` so logs can be saved and checked on PC |
| Acceptance log file save | `Save Log` uses Android `ACTION_CREATE_DOCUMENT` and `openOutputStream` to write the phone log as text |
